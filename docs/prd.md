# Product Requirements Document

## Document Control

- Product: Identity Access Service (IAM)
- Status: Active working draft
- Version: 1.0
- Last updated: 2026-03-30

## Product Summary

Identity Access Service is a backend platform component for SaaS applications that centralizes authentication, role-based authorization, tenant isolation, and auditability. It is intended to behave like a production service rather than a tutorial sample: versioned APIs, database-backed state, migration discipline, operational health checks, and a delivery path that can be extended into a real system.

## Problem Statement

Multi-tenant SaaS systems need a reliable control plane for answering three questions:

1. Who is this actor?
2. What is this actor allowed to do?
3. Which tenant boundary constrains that access?

Many simple examples solve only one of these concerns. This project must solve all three together with operational and engineering rigor.

## Goals

- Provide secure user authentication with short-lived access tokens and revocable refresh tokens.
- Enforce role-based access control across system and tenant scopes.
- Guarantee tenant-aware data access patterns and prevent cross-tenant reads/writes.
- Support tenant-scoped user lifecycle management by privileged administrators.
- Produce auditable records for security-sensitive actions.
- Ship as a containerized, documented, testable backend service with CI.

## Non-Goals

- No frontend application in the initial release.
- No SSO/SAML/OIDC federation in the initial release.
- No fine-grained policy engine beyond RBAC in the initial release.
- No distributed event bus or microservice decomposition in the initial release.
- No billing, subscriptions, or tenant provisioning workflows beyond core CRUD.

## Users and Personas

### System Admin

Responsibilities:

- Create and manage tenants
- Assign high-privilege roles
- Monitor platform state
- Access cross-tenant administrative capabilities

### Tenant Admin

Responsibilities:

- Create and manage users within their tenant
- Assign allowed tenant-scoped roles
- Review tenant-local user listings and activity

### User

Responsibilities:

- Authenticate
- Maintain a valid session
- Access only resources permitted by role and tenant membership

## Core Functional Requirements

### 1. Authentication

Must support:

- User registration for allowed onboarding flows
- Login with tenant context, email, and password
- JWT access token issuance
- Refresh token issuance, rotation, and revocation
- Password hashing with a modern one-way algorithm
- Logout via refresh token invalidation

Acceptance notes:

- Access tokens are short-lived and stateless.
- Refresh tokens are persisted in hashed form.
- Failed login attempts return safe, non-leaking error messages.

### 2. Authorization (RBAC)

Supported roles:

- `SYS_ADMIN`
- `TENANT_ADMIN`
- `USER`

Must support:

- Role-protected routes
- Role checks in service/business logic, not only at the router layer
- Clear distinction between system-wide and tenant-scoped privileges

Authorization rules:

- `SYS_ADMIN` may perform cross-tenant administrative actions.
- `TENANT_ADMIN` may manage users only inside their own tenant.
- `USER` may access only self-service and explicitly permitted resources.

### 3. Multi-Tenancy

Must support:

- Tenant entity with stable identifier and slug
- Every user mapped to exactly one tenant in v1
- Tenant-scoped queries for all tenant-owned resources
- Explicit prevention of cross-tenant access

Design decision:

- Email uniqueness is enforced per tenant, not globally.
- Authentication therefore requires tenant context during login.

### 4. User Management

Must support:

- Create user
- List users within a tenant
- Delete or deactivate user
- Change user role within allowed boundaries
- Retrieve current user profile

Constraints:

- Tenant admins cannot assign or elevate to `SYS_ADMIN`.
- Users cannot manage entities outside their tenant.

### 5. Audit Logging

Must record at minimum:

- Login success
- Login failure
- User creation
- User deletion or deactivation
- Role change
- Refresh token revocation
- Tenant creation

Audit entries should capture:

- Action type
- Actor
- Tenant context
- Target resource
- Timestamp
- Optional metadata such as IP/user agent when available

### 6. Health and System Endpoints

Must support:

- `/api/v1/health` endpoint
- Application version and environment exposure
- Database connectivity status

## API Design Requirements

- RESTful JSON API
- Versioned route namespace under `/api/v1`
- OpenAPI schema automatically generated by FastAPI
- Consistent response and error formats
- Proper HTTP status codes for validation, auth, permission, and not-found errors

## Security Requirements

- Passwords must never be stored or logged in plaintext.
- Refresh tokens must be stored hashed.
- Access to privileged endpoints must require authenticated identity plus role check.
- Multi-tenant access control must be enforced server-side, never trusted from client claims alone.
- Secrets must come from environment variables, not source control.
- Sensitive actions must emit audit records.

## Non-Functional Requirements

- Dockerized local environment
- PostgreSQL persistence
- Alembic migrations
- Clean code organization by domain and layer
- Baseline automated tests
- GitHub Actions CI for lint and test execution
- Readable operational documentation
- Reasonable readiness for extension into production hardening

## Success Criteria

- A new developer can clone the repo, start the stack with Docker Compose, and access docs and health endpoints in under 10 minutes.
- Auth and user-management flows are covered by automated tests with meaningful positive and negative cases.
- Tenant isolation violations are demonstrably prevented in code and tests.
- Sensitive administrative actions are recoverable through audit logs.

## Risks and Open Issues

- Token lifecycle design needs careful attention to rotation and replay safety.
- Tenant-scoped email uniqueness requires a login contract that always includes tenant context.
- Role changes and deletions must avoid orphaned sessions and stale refresh tokens.
- Audit logging should not silently fail in ways that hide security-relevant actions.
